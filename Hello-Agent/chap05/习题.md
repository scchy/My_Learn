## 习题 1

本章介绍了三个各具特色的低代码平台：Coze、Dify 和 n8n。请分析：

1. 这三个平台在**核心定位和设计理念**上有什么区别？它们分别解决了智能体开发中的哪些痛点？

| 维度       | **Coze**      | **Dify**       | **n8n**           |
| :------- | :------------ | :------------- | :---------------- |
| **核心定位** | **C端智能体搭建平台** | **LLM应用开发引擎**  | **工作流自动化编排**      |
| **设计理念** | "让普通人也能做Bot"  | "开发者友好的LLM中间件" | "开源版Zapier/IFTTT" |
| **首要用户** | 运营人员、内容创作者    | 全栈开发者、AI工程师    | 工程师、DevOps、自动化需求者 |

- Coze：降低"创造智能体"的门槛
- Dify：解决"LLM应用工程化"的复杂度
- n8n：打通"跨系统数据流"的断点


2. 低代码平台与纯代码开发各有优劣，此外，也有部分功能用平台实现，部分功能用代码实现的**"混合开发"模式**。思考三种开发模式分别适合哪些场景？请举例说明。


```
1. 主要用户是非技术人员？
   ├─ 是 → Coze（国内生态）或类似Voiceflow
   └─ 否 → 2

2. 核心痛点是"LLM能力编排"？
   ├─ 是 → Dify（RAG/Agent/Workflow一体化）
   └─ 否 → 3

3. 主要是"系统间数据/事件流转"？
   ├─ 是 → n8n（自动化编排最强）
   └─ 否 → 考虑纯代码或更垂直方案
```

## 习题 3

在 5.3 节的 Dify 案例中，我们构建了一个功能全面的"超级智能体个人助手"。请深入分析：

1. 案例中使用了**"问题分类器"**进行智能路由，将不同类型的请求分发到不同的子智能体。这种**多智能体架构**有什么优势？如果不使用分类器，而是让一个单一的智能体处理所有任务，会遇到什么问题？
   1. 职责隔离降低复杂度，每个子智能体只需**加载特定领域工具与Prompt**，减少上下文噪音；故障隔离避免单点崩溃；便于独立迭代优化。
   2. 单一智能体上下文会过长，同时不同类型的问题混杂可能导致混乱，工具膨胀导致选择困难；多领域指令冲突；上下文超载；任一模块改动需全量重测。
2. 数据查询模块需要为大模型提供清晰的**表结构信息**。如果数据库有 50 张表、每张表有 20 个字段，直接将所有 DDL 语句放入提示词会导致上下文过长。请设计一个更智能的方案来解决这个问题。
   1. 按需检索，而非全量堆砌。 
      1. 如果表不多可以先直接人工分类，然后再模糊匹配，分批加载
      2. 也可以意图识别+RAG返回TopK相关表

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  用户Query   │ → │  意图识别   │ → │  表检索引擎  │
│ "上月销售额"  │    │  (关键词/   │    │ (向量相似度/ │
└─────────────┘    │  实体抽取)   │    │  元数据索引) │
                   └─────────────┘    └──────┬──────┘
                                              ↓
                              ┌─────────────────────────┐
                              │  仅返回Top-K相关表(3-5张) │
                              │  + 关键字段 + 样本数据    │
                              └─────────────────────────┘
```

3. Dify 支持**本地部署**和**云端部署**两种模式。请对比这两种模式在数据安全、成本、性能、维护难度等方面的差异，并说明各自适用的场景。

| 维度       | 本地部署            | 云端部署                 |
| :------- | :-------------- | :------------------- |
| **数据安全** | 数据不出境，自主可控      | 依赖云服务商合规，存在传输/存储风险   |
| **成本**   | 硬件+运维人力成本高，无订阅费 | 按需付费，前期投入低，规模扩大后费用累积 |
| **性能**   | 内网延迟低，可定制优化     | 受网络波动影响，高并发需买高配实例    |
| **维护难度** | 需自建团队处理升级、备份、故障 | 托管运维，开箱即用            |

| 模式 | 场景                           |
| :- | :--------------------------- |
| 本地 | 金融/政务/医疗等强合规；已有IDC资源；核心数据敏感  |
| 云端 | 快速验证POC；中小团队无运维人力；流量波动大需弹性伸缩 |

## 习题 5

**提示词工程**在低代码平台中同样至关重要。本章展示了多个平台的提示词设计案例。请分析：

1. 对比 5.2.2 节（Coze）、5.3.2 节（Dify）和 5.4.4 节（n8n）中的提示词设计，它们在**结构、风格和侧重点**上有什么不同？这些差异是否与平台特性相关？

| 维度       | **Coze (5.2.2)**      | **Dify (5.3.2)**                                      | **n8n (5.4.4)**                              |
| -------- | --------------------- | ----------------------------------------------------- | -------------------------------------------- |
| **结构**   | 角色定义 + 工作流 + 注意事项     | Profile + Skills + Rules + Workflows + Initialization | System Message + 上下文信息 + 可用工具 + 执行步骤 + 输出格式  |
| **风格**   | 指令式、任务导向              | 能力导向、规范约束                                             | 工程化、流程化、强约束                                  |
| **侧重点**  | **内容生成质量**（筛选、整合、去重）  | **交互体验**（友好、耐心、可解释）                                   | **系统集成**（工具调用、数据格式、状态管理）                     |
| **变量使用** | `{{articles}}` 等数据占位符 | 无（纯角色定义）                                              | `{{ $json.From }}`、`{{ new Date() }}` 等动态表达式 |
| **输出控制** | 数量约束（10条新闻、5篇论文、5个项目） | 软性约束（适度建议、通俗易懂）                                       | 硬性格式约束（JSON结构、`<br>`标签、布尔标志）                 |

三平台的提示词设计差异本质上是产品形态差异的映射：
- Coze = 内容策展（Curator）→ 关注信息质量与呈现
- Dify = 对话伙伴（Companion）→ 关注体验与可解释性
- n8n = 自动化引擎（Engine）→ 关注确定性执行与系统集成


2. 在 Dify 的"文案优化模块"中，提示词要求输出"**超过 500 字**"。这种对输出长度的硬性要求是否合理？
   - 在什么情况下应该限制输出长度？
   - 什么情况下应该让模型自由发挥？

| 场景          | 是否应该硬性限制 | 理由                     |
| ----------- | -------- | ---------------------- |
| **SEO长文生成** | ✅ 合理     | 搜索引擎偏好长内容，硬性下限确保基础质量   |
| **商品详情页**   | ✅ 合理     | 电商平台有最低字数要求以提升信息密度     |
| **学术论文辅助**  | ⚠️ 需谨慎   | 应限制的是"论证深度"而非字数，避免灌水   |
| **创意写作/诗歌** | ❌ 不合理    | 限制字数会扼杀创意，应自由发挥        |
| **技术文档**    | ❌ 不合理    | 应追求"清晰简洁"，强制500字可能导致冗余 |

## 习题 7

**平台选型**是智能体产品成功的关键决策之一。假设你是一家初创公司的技术负责人，公司计划开发以下三个 AI 应用，请为每个应用选择最合适的平台（Coze、Dify、n8n 或纯代码开发），并详细说明理由：

| 应用 | 描述 |
|------|------|
| **应用 A** | 面向 C 端用户的"AI 写作助手"小程序，需要快速上线验证市场需求，预算有限，团队中只有 1 名前端工程师和 1 名产品经理 |

> - 技术可行性

> - 成本控制
> - 可维护性
> - 可扩展性
> - 数据安全与合规性
> - 平台能力是否满足需求
> - 上线速度
> - 运营成本
> - 后续迭代难度
> - 未来功能扩展空间
