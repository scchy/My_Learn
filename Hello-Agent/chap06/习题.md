本章介绍了四个各具特色的智能体框架：AutoGen、AgentScope、CAMEL 和 LangGraph。请分析以下问题：

### 1.1 框架对比

在 6.1.2 节的表 6.1 中，对比了这四个框架的多个维度。请选择其中两个你最熟悉的框架，从以下三个维度进行深入对比：

- **协作模式**
- **控制方式**
- **适用场景**  

| 维度       | AutoGen                                                                                                           | LangGraph                                                                                                    |
| :------- | :---------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------- |
| **协作模式** | **对话驱动（Conversation-Driven）**<br>将任务执行抽象为多角色参与的"群聊"。智能体通过自然语言对话进行协作，如产品经理→工程师→审查员的软件开发流程。协作行为从角色定义和对话规则中**自然涌现**。 | **状态机驱动（State Machine-Driven）**<br>将工作流显式建模为有向图，节点代表计算步骤（LLM调用/工具执行），边定义跳转逻辑。协作通过**预定义的图结构**严格执行，支持循环实现迭代反思。 |
| **控制方式** | **隐式控制**：通过系统消息（System Message）定义角色职责，通过 `RoundRobinGroupChat` 等机制控制发言顺序。控制逻辑分散在对话上下文中，灵活但难以精确预测执行路径。             | **显式控制**：开发者精确定义状态（State）、节点（Node）、边（Edge）及条件跳转。每个步骤都是确定的 Python 函数，支持"人机协作"节点插入，可控性强但牺牲了动态性。                |
| **适用场景** | 适合**开放式、创造性任务**：<br>• 多角色头脑风暴<br>• 需求不明确需反复澄清的项目<br>• 需要"人类在环"（Human-in-the-loop）的协作场景                            | 适合**流程化、高可靠性任务**：<br>• 需要精确步骤控制的业务流程<br>• 需审计、可复现的生产级应用<br>• 包含"反思-修正"循环的复杂工作流                               |


### 1.2 设计哲学理解

本章提到了"涌现式协作"与"显式控制"之间的权衡，请解释这两种设计哲学的含义。

涌现式协作：  通过定义智能体的角色、目标和简单交互规则，让复杂的协作行为从系统中自发"生长"出来，而非被显式编码。  
  - 代表框架：AutoGen（群聊模式）、CAMEL（角色扮演）  

显式控制：开发者精确定义智能体执行的每一个步骤、状态转移条件和循环逻辑，将工作流固化为可验证的结构。   
 - 代表框架：LangGraph（状态机）、传统工作流引擎

| 维度          | 涌现式协作       | 显式控制         |
| :---------- | :---------- | :----------- |
| **开发速度**    | 快（定义角色即可启动） | 慢（需完整建模流程）   |
| **运行时可预测性** | 低           | 高            |
| **容错能力**    | 强（可协商恢复）    | 弱（需预设错误处理分支） |
| **适用任务类型**  | 探索性、创造性     | 流程化、事务性      |


选择原则：
- 任务边界模糊、需要多方协商 → 涌现式协作（AutoGen/CAMEL）
- 任务步骤明确、需要严格合规 → 显式控制（LangGraph）

### 3.1 消息驱动架构

案例中使用了 `MsgHub`（消息中心）来管理智能体间的通信。请回答：

- 消息驱动架构相比传统函数调用的优势是什么？  

| 维度       | **传统函数调用**        | **消息驱动架构 (MsgHub)**     |
| -------- | ----------------- | ----------------------- |
| **耦合度**  | 强耦合（调用方需知道被调方的接口） | 松耦合（只关心消息格式，不关心接收者）     |
| **扩展性**  | 修改调用链需改动多处代码      | 新增智能体只需订阅/发布消息，无需改动现有逻辑 |
| **并发模型** | 同步阻塞，调用栈深度随链路增长   | 异步非阻塞，天然支持并行处理          |
| **可观测性** | 需手动埋点追踪调用链        | 消息即日志，天然可追踪、可回放         |
| **容错性**  | 单点失败可能导致整个调用链崩溃   | 消息可持久化、重试，失败隔离          |
| **动态性**  | 编译时确定调用关系         | 运行时动态订阅/取消，支持热插拔        |

- 在什么场景下这种架构特别有价值？
  - 多智能体动态组网：狼人杀-角色身份、阵营、生死状态动态变化，硬编码关系无法适应
  - 人机混合协作：AI玩家+人类玩家同台-人类通过UI发布/订阅消息，与AI统一接口，无需特殊适配层 

> 消息驱动架构将"智能体间的关系"从代码中解放出来，转变为可配置、可观测、可回放的运行时数据，这是多智能体系统从"硬编码脚本"走向"涌现式协作"的关键基础设施。


### 3.2 结构化输出设计

游戏中使用了结构化输出（如 `DiscussionModelCN`、`WitchActionModelCN`）来约束智能体行为。请设计一个新的游戏角色**猎人**，并定义其对应的结构化输出模型，包括：

- 字段定义
- 验证规则

```python
from pydantic import BaseModel, Field, model_validator
from typing import Optional, Literal
from enum import Enum


class DeathCause(str, Enum):
    """死亡原因枚举"""
    WOLF_KILL = "wolf_kill"           # 被狼人杀死
    VOTE_EXECUTION = "vote_execution" # 被投票处决
    POISONED = "poisoned"             # 被女巫毒死
    LOVER_DEATH = "lover_death"       # 殉情（若游戏有情侣机制）
    UNKNOWN = "unknown"               # 未知/其他


class HunterTriggerModelCN(BaseModel):
    """
    猎人死亡触发技能时的决策模型
    核心决策：是否开枪 + 带走谁 + 策略考量
    """
    
    # ========== 基础判断 ==========
    
    can_shoot: bool = Field(
        description="当前死亡情境下是否能够开枪（被毒死则不能）",
        default=True
    )
    
    death_cause: DeathCause = Field(
        description="死亡原因，决定技能是否可用",
        default=DeathCause.UNKNOWN
    )
    
    # ========== 核心决策 ==========
    
    will_shoot: bool = Field(
        description="【关键决策】是否发动技能开枪",
        default=False
    )
    
    target_name: Optional[str] = Field(
        description="若开枪，目标玩家姓名（必须为非空有效玩家名）",
        default=None,
        min_length=1,
        max_length=20
    )
    
    shoot_reason: Optional[str] = Field(
        description="开枪/不开枪的策略理由（用于复盘与可解释性）",
        default=None,
        max_length=200
    )
    
    # ========== 策略评估 ==========
    
    confidence_level: int = Field(
        description="对目标身份判断的信心程度(1-10)",
        ge=1, le=10,
        default=5
    )
    
    target_suspected_role: Optional[Literal["werewolf", "villager", "unknown"]] = Field(
        description="推测目标的身份（狼人/平民/未知）",
        default="unknown"
    )
    
    # ========== 备选方案（不开枪时的策略） ==========
    
    alternative_strategy: Optional[str] = Field(
        description="若选择不开枪，后续轮次的策略考虑",
        default=None,
        max_length=150
    )

    # ========== 验证规则 ==========
    
    @model_validator(mode='after')
    def validate_shooting_logic(self):
        """
        核心验证逻辑：
        1. 被毒死时不能开枪
        2. 开枪时必须指定目标
        3. 不开枪时目标应为空
        4. 信心程度与开枪决策的合理性检查
        """
        
        # 规则1：被毒死强制不能开枪
        if self.death_cause == DeathCause.POISONED:
            if self.can_shoot:
                raise ValueError("被女巫毒死时，can_shoot 必须为 False")
            if self.will_shoot:
                raise ValueError("被女巫毒死时无法开枪，will_shoot 必须为 False")
        
        # 规则2：开枪时必须指定目标
        if self.will_shoot:
            if not self.target_name or self.target_name.strip() == "":
                raise ValueError("决定开枪时必须指定 target_name")
            if not self.shoot_reason or len(self.shoot_reason) < 5:
                raise ValueError("开枪时必须提供充分的 shoot_reason（至少5个字符）")
        
        # 规则3：不开枪时不应有目标
        if not self.will_shoot and self.target_name is not None:
            raise ValueError("决定不开枪时，target_name 应为 None")
        
        # 规则4：信心程度与开枪决策的一致性
        if self.will_shoot and self.confidence_level < 3:
            raise ValueError("信心程度低于3时开枪风险过高，建议不开枪或重新评估")
        
        return self

    class Config:
        json_schema_extra = {
            "example": {
                "can_shoot": True,
                "death_cause": "vote_execution",
                "will_shoot": True,
                "target_name": "张三",
                "shoot_reason": "张三白天发言漏洞明显，且投票阶段行为与狼人一致，带走可为民排坑",
                "confidence_level": 7,
                "target_suspected_role": "werewolf",
                "alternative_strategy": None
            }
        }
```

### 4.1 冲突解决机制

在案例中，协作会在检测到 `<CAMEL_TASK_DONE>` 标志时强制终止。但如果两个智能体意见分歧（一位认为可以终止，一位认为不应该终止），无法达成一致怎么办？请设计一个**冲突解决**的兼容机制。

> 引入双方确认 + 分歧仲裁
> 下面是核心函数，详见[play_camel.py](./play_camel.py)

```python
task_prompt = """
创作一本关于"拖延症心理学"的短篇电子书，目标读者是对心理学感兴趣的普通大众。
要求：
1. 内容科学严谨，基于实证研究
2. 语言通俗易懂，避免过多专业术语
3. 包含实用的改善建议和案例分析
4. 篇幅控制在8000-10000字
5. 结构清晰，包含引言、核心章节和总结

【重要】当你认为任务已完成时，回复中必须包含 <CAMEL_TASK_DONE> 标志。
当你认为对方提出的完成提议不合理时，回复中必须包含 <CAMEL_TASK_REJECT> 标志并说明理由。
"""


def check_termination_status(assistant_msg: str, user_msg: str) -> tuple:
    """
    检测双方终止意图
    返回: (assistant_done, user_done, conflict_detected, reject_reason)
    """
    assistant_done = "CAMEL_TASK_DONE" in assistant_msg
    user_done = "CAMEL_TASK_DONE" in user_msg
    assistant_reject = "CAMEL_TASK_REJECT" in assistant_msg
    user_reject = "CAMEL_TASK_REJECT" in user_msg

    # 冲突情况：一方想终止，另一方明确反对
    conflict = (assistant_done and user_reject) or (user_done and assistant_reject)
    # 提取拒绝理由（简单正则）
    reject_reason = ""
    if conflict:
        reject_match = re.search(r'<CAMEL_TASK_REJECT>(.*?)(?=<|$)', 
                                assistant_msg if user_done else user_msg, 
                                re.DOTALL)
        if reject_match:
            reject_reason = reject_match.group(1).strip()[:200]  # 限制长度
    
    return assistant_done, user_done, conflict, reject_reason



def resolve_conflict(assistant_msg: str, user_msg: str, turn: int, max_turns: int) -> str:
    """
    极简仲裁：基于回合数和信心度做决策
    """
    assistant_conf = _extract_confidence(assistant_msg)
    user_conf = _extract_confidence(user_msg)
    
    # 规则1：接近上限，强制终止
    if turn >= max_turns - 3:
        return "force_terminate"
    
    # 规则2：高信心方胜（信心差距>2）
    if abs(assistant_conf - user_conf) > 2:
        return "terminate" if max(assistant_conf, user_conf) == assistant_conf else "continue"
    
    # 规则3：保守策略，默认继续
    return "continue"


def _extract_confidence(msg: str) -> int:
    """从消息中提取信心值（1-10），默认5"""
    match = re.search(r'信心[:：]\s*(\d+)', msg)
    if match:
        return min(10, max(1, int(match.group(1))))
    # 通过关键词估算
    strong_words = ['完成', '达成', '满足', '符合']
    weak_words = ['需要', '改进', '不足', '缺少']
    score = 5 + sum(1 for w in strong_words if w in msg) - sum(1 for w in weak_words if w in msg)
    return min(10, max(1, score))
```


### 4.2 多智能体协作

CAMEL 最初设计用于双智能体协作，但现在已经扩展支持多智能体。请查阅 CAMEL 的最新文档，了解其多智能体协作模块 `workforce`，并结合架构图说明其与 AutoGen 的群聊模式有何不同。
> 本质区别：AutoGen 是"对话驱动"（Conversation-Driven），强调自由交流；CAMEL Workforce 是"任务驱动"（Task-Driven），强调层级调度与自动容错

架构
```
┌─────────────────────────────────────────┐
│         Workforce (根节点)               │
│  ├─ Coordinator Agent (协调器)          │
│  └─ Task Planner Agent (任务规划器)      │
├─────────────────────────────────────────┤
│         Worker Nodes (工作节点)           │
│  ├─ SingleAgentWorker (单智能体)         │
│  ├─ RolePlayingWorker (双智能体角色扮演)  │
│  └─ Sub-Workforce (嵌套Workforce)        │
│       └─ 支持递归分层                    │
└─────────────────────────────────────────┘
```
差异枚举

| 维度       | **AutoGen GroupChat**        | **CAMEL Workforce**       |
| -------- | ---------------------------- | ------------------------- |
| **拓扑结构** | 扁平群聊（任意Agent可发言）             | 分层树形（Coordinator集中调度）     |
| **通信模式** | 广播式（所有消息全员可见）                | 通道式（Task Channel定向分发）     |
| **任务分配** | 基于发言意愿/轮询（Speaker Selection） | 基于能力匹配（Coordinator智能指派）   |
| **动态扩展** | 需预定义所有Agent                  | 运行时动态创建Worker节点           |
| **失败处理** | 依赖对话恢复或人工介入                  | 自动分解/重组/创建新Worker         |
| **层级深度** | 单层群聊                         | 支持Workforce嵌套（递归分层）       |
| **人机交互** | UserProxyAgent嵌入对话           | 支持Human-in-the-Loop但不打断结构 |
| **适用场景** | 头脑风暴、开放式讨论                   | 复杂任务自动化、工作流执行             |

def reflect_node(state: SearchState) -> SearchState:
    """
    步骤4：质量反思与评估
    输出: quality_score + retry_strategy
    """
    
    answer = state.get("final_answer", "")
    search_results = state.get("search_results", "")
    reflection_count = state.get("reflection_count", 0)
    
    # 硬约束：最多反思2次，防止无限循环
    if reflection_count >= 2:
        print(f"⏹️ 已达最大反思次数({reflection_count})，强制结束")
        return {
            "quality_score": 0.6,  # 中等评分
            "retry_strategy": "continue",
            "step": "reflected",
            "reflection_count": reflection_count,
            "messages": [AIMessage(content="[系统：已达到质量检查上限，返回当前最佳答案]")]
        }
    
    # 构建反思提示
    reflect_prompt = f"""评估以下答案质量：

【用户问题】{state['user_query']}
【搜索结果长度】{len(search_results)} 字符
【生成答案】{answer[:800]}...

请从以下维度评分（0-1）：
1. 完整性：是否充分回答问题？
2. 具体性：是否包含细节、数据、案例？
3. 引用度：是否基于搜索结果？
4. 长度：是否过于简短（<150字）？

输出格式：
质量评分：[0-1之间的数字]
主要问题：[简短描述最大缺陷]
改进建议：[如何改进]
是否重新搜索：[是/否]
是否重新生成：[是/否]"""

    reflection = llm.invoke([SystemMessage(content=reflect_prompt)])
    reflection_text = reflection.content

    # 解析反思结果
    quality_score = 0.5
    if "质量评分：" in reflection_text:
        try:
            score_str = reflection_text.split("质量评分：")[1].split()[0]
            quality_score = float(score_str)
        except:
            pass

    # 确定重试策略
    retry_strategy: Literal["re_search", "re_generate", "continue"] = "continue"
    
    # 条件逻辑
    if quality_score < 0.6:
        if "重新搜索：是" in reflection_text or len(search_results) < 200:
            retry_strategy = "re_search"
        elif "重新生成：是" in reflection_text:
            retry_strategy = "re_generate"
    
    # 提取反馈用于下一轮
    feedback = ""
    if "改进建议：" in reflection_text:
        feedback = reflection_text.split("改进建议：")[1].split("\n")[0][:100]

    print(f"🤔 反思结果: 评分={quality_score:.2f}, 策略={retry_strategy}, 次数={reflection_count+1}")
    
    return {
        "quality_score": quality_score,
        "retry_strategy": retry_strategy,
        "reflection_feedback": feedback,
        "reflection_count": reflection_count + 1,
        "step": "reflected",
        "messages": [AIMessage(content=f"[质量检查: {quality_score:.0%}]")]
    }


# ========== 条件边路由函数 ==========

def route_by_quality(state: SearchState) -> Literal["re_search", "re_generate", "end"]:
    """
    条件边：根据反思结果决定下一步
    """
    strategy = state.get("retry_strategy", "continue")
    score = state.get("quality_score", 0.5)
    
    if strategy == "re_search":
        return "re_search"
    elif strategy == "re_generate":
        return "re_generate"
    else:
        return "end"
